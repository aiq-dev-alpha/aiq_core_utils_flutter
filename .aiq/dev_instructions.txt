  :: dsl_notation
    // the goal of this is notation is to standarize how we can use indentation and tags to represent complex context and preserve better meaning for ai to read and for humans.
    // the language also promoptes low token usage and true content generation that a human can easily read.

      :- title_markers
          - ::        domain heading (top level)
          - ::-       sub-heading (under domain)
          - :-        sub-heading sub-heading (under sub-heading)

      :- heading_markers
          - ::        domain heading (top level)
          - ::-       sub-heading (under domain)
          - :-        secondary sub-category (indented 4 spaces)
          - -::       bullet point that is a domain
          - -         sub domain bullet point
          - -         bullet point

      :- tag_markers
        // tags are added generally under the title to describe the sections of the document and document even better then the title
        // tags provide better context for searching

          - ::=       domain level tags
          - :=        item level tags

      :- special_markers
          - :!        ai todo item
          - `{code}`  inline code
          - ```{code}``` multi-line code block


      :- indentation_rules
          - 4 spaces per indent level
          - never use actual tabs
          - items under title: 4 spaces
          - items under subtitle: 8 spaces                                                                                            
          - child content under a bullet uses `- ` at next indent level                                        
          - bare lines without `- ` prefix are never valid under a `- ` item      
            - only special output contents at final lower level may not use dashes but indentation only

      :- comments
          //            user single line comments

      :- output_rules
          - never use ====, ----, ****, or any decorative line separators — structure comes from indentation and markers only
          - ai must use this dsl notation when outputting any content for the user to read
          - for any non code output ai must minimize output as much as possible to save tokens and prevent context from being full
          - all plans, phases, steps, tasks, tracking files, and documentation use this format
          - never output all caps section headers
          - never output plain markdown headers for aiq content
          - only output the relevant :: section, never the full document
          - user identifies sections by :: [domain] — never repeat surrounding context and always show the full :: [domain] indented section
          - minimize tokens on all non-code output
          - never output all caps section headers or plain markdown headers for aiq content
          - underscore in all names and file names, never dash
          - for non-code output minimize tokens as much as possible

  :: offline_modes
  ::= three_distinct_modes, never_confuse_them

      :- offline_first (production behaviour)
          - every app stores data locally first and syncs to cloud when online
          - data persists across installs and app restarts
          - app is fully functional with no internet connection
          - this is the default for every app unless user explicitly says otherwise

      :- offline_dev_test_mode (dev only — fully stripped at uat and prod)
          - controlled by `--dart-define=ENV=dev|uat|prod` at build time
          - kDebugMode is not used — env dart-define is the only build variant flag
          - String.fromEnvironment is a compile-time constant — if (Env.isOfflineDevTest)
            blocks are fully dead code at uat/prod and tree-shaken out completely
          - the env class must look exactly like this in every app, no variations:

              ```dart
              class Env {
                  static const String _env = String.fromEnvironment('ENV', defaultValue: 'dev');
                  static bool get isOfflineDevTest => _env == 'dev';
                  static bool get isUat => _env == 'uat';
                  static bool get isProd => _env == 'prod';
              }
              ```

          - every service call, repository access, and auth flow must follow this exact pattern:

              ```dart
              if (Env.isOfflineDevTest) {
                  // offline test logic — uses OfflineTestDataRepository
                  // fake service calls appear in dev_http_inspector like real calls
                  // this entire block is gone at uat/prod
              } else {
                  // normal online logic — uses real repository and real backend
              }
              ```

          - i want the ability to turn offline_dev_test_mode have the ability to hide and show fully using a boolean variable that can be controlled with ai using project-info.txt - this way i can build a dev build without offline_dev_test_mode 
          - no other pattern is acceptable — never mix offline logic into online paths
          - uses json files in test_data/ folder — never touches real backend
          - fake service calls must appear in dev_http_inspector exactly like real calls
          - user can log in offline and fully navigate the app as if online
          - user can switch between multiple mock users cleanly via debug_ui_panel —
            each switch must feel like a fresh app launch with no state bleed from previous user
          - OfflineTestDataRepository and the real repository both extend the same base
            repository class — changes to the interface propagate to both automatically

      :- offline_safe_data_mode (dev only — available only inside offline_dev_test_mode)
          - uses json files in safe_test_data/ folder instead of test_data/
          - contains sfw content replacements — for working in public or screen sharing
          - separate from the nsfw blur/placeholder system which is a runtime user toggle
          - switching to safe data mode replaces data source only — all app logic stays identical
          - only available when ENV=dev and isOfflineDevTest is active

  :: plan_format
      plan -> phase -> step -> task

      :- plan rules
          - one user request = one plan
          - groups all phases needed to satisfy the request end to end
          - no info variable — plan is the top level container

      :- phase rules
          - one focused area of work within a plan
          - runs in order, 3 passes before done
          - max 4 steps per phase — if more are needed, create a _b phase for overflow
          - if a phase touches more than 20 files or exceeds 60 minutes, split it
          - phase_info: 4 to 5 paragraphs — a cold agent reading only this block must fully
            understand the user's original request, what has been built so far, what this
            phase delivers, and what done looks like — user context must never be compressed
            or lost across a sequence of phases

      :- step rules
          - every step line in dev_plan.txt, dev_overall_tasks.txt, and dev_current_tasks.txt                            
                must carry 2-4 sentences of inline detail — never just a short label like
                "single-field settings forms" — the step line itself must describe what files
                are touched, what pattern is being applied, and how many items are in scope

          - logical grouping of tasks within a phase
          - max 8 tasks per step

              2-4 sentence detail from dev_plan.txt — never abbreviate to one-liners like

          - single unit of work, bottom level, no info variable
          - never a one-liner — must have enough inline detail that output would not be
            garbage if the agent had only that task to read
          - vague tasks like "improve performance" or "fix auth" are not allowed
          - if context is lost and output would be garbage, the task is written wrong

        :- app_section_format                                                                                                              
            - in multi-project workspaces, dev_plan.txt, dev_plan_all_archive.txt,
              dev_overall_tasks.txt, and dev_current_tasks.txt are grouped by app
            - top level marker: [app: app_name] (e.g., [app: project_hearts])
            - use [app: shared] for cross-app work (aiq_core_, dev_ plugins, workspace-level)
            - each [app: app_name] section is completely independent from all other app sections
            - ai must never mix content from one app section into another
            - reading one app section must give full context without needing other sections

        :- per_app_numbering
            - every counter is per app — phases, steps, task groups all scoped to [app: app_name]
            - each app starts at tasks_1 and increments sequentially (tasks_1, tasks_2, ... tasks_77, tasks_200)
            - phase numbers also start at phase_1 per app
            - [app: shared] has its own independent counters starting at 1
            - no number is shared or coordinated across apps
            - prompt_counter remains single global counter (not per app)

            - dev_current_tasks.txt has one [app: app_name] block per active app
            - only apps with active work appear in dev_current_tasks.txt
            - dev_plan.txt has all apps, each under their own [app: app_name] heading
            - dev_overall_tasks.txt mirrors dev_plan.txt structure per app
            - dev_plan_all_archive.txt archives per app, never merge across apps


  :: phase_execution
      - in dev_plan all phases must be completed one after another
      - do not start a new phase until the current one is completed
      - work phase by phase, step by step, task by task
      - 3 passes required per phase before marking done, passes are not repetitions:
          - pass_1: implement and get it working in offline/demo mode
          - pass_2: harden — security, edge cases, type errors, no debug leaks
          - pass_3: optimize — bundle, lazy loading, memory, no regressions
      - each pass is distinct work with its own commit tagged [PASS1] [PASS2] [PASS3]
